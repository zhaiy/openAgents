# OpenAgents 下一期迭代计划

> 创建日期：2026-03-25
> 适用阶段：上一期“恢复与观测”完成之后的下一期
> 规划主题：复盘与演进（聚焦版）

---

## 一、迭代定位

上一期已经重点完成以下方向：

- 节点级恢复能力
- 恢复预览与风险提示
- 成本观测能力
- 质量观测能力
- 恢复与观测相关测试补强

这意味着 OpenAgents 现在已经不只是“能运行、能看状态”，而是开始积累：

- run 与 recovered run 的关系
- 成本与质量指标
- diagnostics / compare / recovery 的操作上下文

下一期不建议马上继续横向扩功能，而应优先把这些新增数据沉淀成真正可用于调优决策的复盘能力：

1. 让用户知道一次 run 对应的是哪一版 workflow、agent、prompt、配置
2. 让用户知道最近几次运行是变好了、变差了，还是出现了回归
3. 让用户在失败后快速获得“发生了什么、影响了什么、下一步看哪里”的复盘摘要

一句话概括：

> 下一期不是继续堆入口，而是把 OpenAgents 从“可恢复、可观测的工作台”推进到“可比较、可复盘、可用于调优决策的运营台”。

---

## 二、迭代目标

### 目标 G1：建立稳定的运行版本上下文

让 run 不再只是一次执行记录，而是可追溯到 workflow、agent、prompt、关键配置版本或快照的历史样本。

### 目标 G2：建立可解释的差异能力

让 compare 不只是“结果不同”，而是能够回答“哪些配置变了、哪些节点变了、这些变化为什么可能影响结果”。

### 目标 G3：建立趋势性复盘能力

把上一期沉淀下来的恢复、成本、质量指标组织成历史趋势，让用户能判断某个 workflow 是在变好、变差，还是开始波动。

### 目标 G4：建立失败复盘闭环

围绕失败 run 提供结构化复盘摘要，把 diagnostics、compare、timeline、recovery 串成更完整的行动链路。

### 目标 G5：继续加固工程回归基础

复盘能力会引入版本快照、差异聚合、历史指标、跨页面联动，这些都必须有自动化测试保护，否则很容易做成表面可看、底层不稳的功能。

---

## 三、本期重点原则

1. 先定义版本快照与指标口径，再做页面表达。
2. 复盘闭环优先于新增页面或新入口。
3. 单次 run 页面必须能回到“版本差异 + 历史趋势”的上下文中。
4. 不提前展开团队协作、复杂资产管理或重报表系统。
5. 每个任务包至少形成 API、页面、测试中的两层闭环。
6. 不通过前端特判去掩盖版本来源不清或统计口径不稳的问题。
7. 不做大规模重构，优先做低风险、可验证、可逐步演进的增强。

## 五、正式任务包

## E1. 运行版本快照与关联基线

### 目标

让每个 run 都能稳定关联到 workflow、agent、prompt、关键配置的版本上下文或快照引用，避免后续 compare 和复盘只能靠页面猜测来源。

### 重点内容

- 定义 run 级版本来源字段与快照引用表达
- 明确 rerun、rerun-with-edits、recover 后的版本承接关系
- 收口 source run、recovered run、compared run 在版本语义上的共同字段
- 让 run detail、compare、diagnostics 能消费同一套版本上下文

### 交付物

- 一版稳定的 run version snapshot / provenance 契约
- 相关 DTO / API / service 实现
- 契约与关联关系测试

### 优先级

`P0`

### 预计收益

- 后续差异分析不再缺少“比较基线”
- 失败 run、恢复 run、重跑 run 的关系更容易解释
- 为趋势分析提供稳定的历史样本维度

---

## E2. 版本差异视图

### 目标

让 compare 从“看结果差异”升级为“解释版本与配置差异”，帮助用户判断结果变化是不是由 workflow、prompt、model、输入结构等变更导致。

### 重点内容

- 支持 workflow 结构、节点配置、模型、提示词、关键参数差异摘要
- 区分“影响执行路径”和“影响输出质量”的差异类型
- 让 compare 页面能显示“本次差异为什么值得关注”
- 为失败 run 或回归 run 提供版本变化上下文

### 交付物

- 版本差异 DTO 与聚合服务
- compare 页面差异摘要区
- 差异契约与页面行为测试

### 优先级

`P0`

### 预计收益

- 用户能更快回答“为什么这次结果不一样”
- compare 更适合做 prompt / workflow 调优
- 为失败复盘提供直接背景

---

## E3. 历史趋势分析

### 目标

让恢复、成本、质量这几类指标从“单次可见”升级为“历史可比较”，形成 workflow 级趋势判断能力。

### 重点内容

- 定义最近 N 次 run 的趋势摘要口径
- 聚合成功率、失败率、耗时、token、gate、recovery 等核心趋势指标
- 提供基础异常波动或回归信号
- 在首页、workflow 概览或 runs 列表中补充趋势摘要入口

### 交付物

- workflow trend DTO 与聚合服务
- 历史趋势摘要卡片或列表增强
- 趋势统计测试

### 优先级

`P1`

### 预计收益

- 更容易看出优化是否真的生效
- 更容易定位“最近开始变差”的 workflow
- 为更深趋势分析和复盘能力提供第一版基线

---

## E4. 失败 run 复盘摘要

### 目标

让失败 run 不再只展示零散状态，而是能生成结构化复盘摘要，帮助用户快速理解失败原因、受影响范围、相关版本变化和可执行下一步。

### 重点内容

- 汇总失败节点、失败类型、下游影响范围
- 结合版本差异、恢复动作、关键指标输出复盘摘要
- 给出后续建议入口，如 compare、diagnostics、recover、rerun-with-edits
- 在 run detail 和 diagnostics 中形成统一复盘表达

### 交付物

- 失败复盘 DTO 与聚合逻辑
- run detail / diagnostics 复盘摘要区
- 复盘摘要测试

### 优先级

`P0`

### 预计收益

- 失败处理从“自己拼信息”升级为“有结构地判断”
- diagnostics 真正成为复盘入口，而不是只看问题列表
- 更容易进入“分析 -> 修复 -> 验证”闭环

---

## E5. 复盘链路联动收口

### 目标

把 compare、diagnostics、timeline、recovery 这些已有能力收成一条更连贯的复盘链路，而不是让用户在多个页面之间自己找入口。

### 重点内容

- 统一失败 run 的入口跳转与推荐动作
- 让版本差异、历史趋势、失败复盘能互相跳转
- 强化 recovered run 与 source run 的复盘联动
- 补足关键文案、空态、异常态和导航闭环

### 交付物

- 跨页面复盘联动方案
- 页面侧入口和导航收口
- 基础 smoke / E2E 测试

### 优先级

`P1`

### 预计收益

- 用户不必自己拼复盘路径
- compare / diagnostics / run detail 的角色边界更清晰
- 复盘效率更高，学习成本更低

---

## E6. 复盘与趋势测试补强

### 目标

把版本快照、差异分析、趋势统计、失败复盘这些新语义纳入自动化回归，确保复盘能力不是脆弱的展示层拼接。

### 重点内容

- 版本快照与 run 关联测试
- 版本差异契约与聚合测试
- 趋势统计口径测试
- 失败复盘摘要测试
- 关键跨页面联动 smoke / E2E 回归

### 交付物

- 一组覆盖复盘主链路的自动化测试
- 一组覆盖趋势与差异口径的统计测试
- 对应的验收清单

### 优先级

`P0`

### 预计收益

- 防止版本来源和差异结果静默漂移
- 防止趋势指标在不同页面出现不一致
- 降低下一轮更深复盘功能的回归风险

---

## E7. 快照与聚合层适度整理

### 目标

在不做大迁移的前提下，适度收敛版本快照、趋势指标、复盘摘要相关聚合逻辑，避免后续迭代再次在多个服务和页面里重复计算。

### 重点内容

- 收敛 run metrics / compare / diagnostics 中重复的聚合逻辑
- 明确“版本快照”“趋势指标”“复盘摘要”三类 DTO 的职责边界
- 清理明显长期无价值的临时兼容字段
- 为后续更深的复盘和趋势能力预留统一入口

### 交付物

- 轻量聚合层整理
- 统计与版本口径说明
- 必要的低风险回归测试

### 优先级

`P2`

### 预计收益

- 后续趋势和复盘扩展更容易落地
- 页面间统计口径更一致
- 控制服务层复杂度增长

---

## 六、本期推荐开发顺序

建议按以下顺序推进：

1. `E1` 运行版本快照与关联基线
2. `E2` 版本差异视图
3. `E4` 失败 run 复盘摘要
4. `E6` 复盘与趋势测试补强（核心部分应伴随 `E1/E2/E4` 同步推进）
5. `E3` 历史趋势分析
6. `E5` 复盘链路联动收口
7. `E7` 快照与聚合层适度整理

关键路径如下：

```text
E1 -> E2 -> E4 -> E5 -> E6
  \-> E3 -----------^
E7 可伴随进行，但不应阻塞主链路
```

---

## 七、本期价值

如果这一期按上述方向完成，OpenAgents 会从“可恢复、可观测的工作台”进一步进入“可比较、可复盘、可持续优化”的阶段。

这期最大的价值不在于新增多少页面，而在于三点：

1. 用户能知道一次 run 背后对应的是哪一版配置和 workflow 上下文
2. 用户能判断某次结果变化是版本变化、指标回归，还是单次异常
3. 用户能更快完成“失败复盘 -> 恢复/调整 -> 再验证”的调优闭环
