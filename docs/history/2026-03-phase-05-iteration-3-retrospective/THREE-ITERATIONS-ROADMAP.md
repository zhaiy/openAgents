# OpenAgents 未来三次迭代路线图

> 创建日期：2026-03-22
> 适用范围：下一期起连续三次迭代
> 规划目标：持续把现有能力做深、做稳、做成体系

---

## 一、路线图定位

接下来的三次迭代，不以“快速扩张功能面”为主，而以“夯实已有系统”为主。

核心判断如下：

1. OpenAgents 已完成从 CLI 到 Web Workbench 的第一轮形态升级
2. 当前最需要解决的问题不是“功能入口不足”，而是“系统是否稳定、一致、可恢复、可复盘”
3. 如果当前阶段继续横向堆功能，后续维护成本、联调成本、回归风险都会快速上升

因此，未来三期建议围绕同一条主线推进：

> 先做稳，再做深，最后做成可持续优化的平台能力。

---

## 二、三期总目标

### 总目标 T1：把现有核心链路打造成稳定产品能力

重点围绕：

- 配置运行
- 实时观察
- Gate 处理
- 失败诊断
- 重跑与恢复
- 结果复盘

### 总目标 T2：把流程引擎与 Web Workbench 的数据边界做清楚

避免后续每加一个能力，都要同时返工服务层、状态层、前端页面和测试。

### 总目标 T3：逐步建立“可恢复、可观测、可比较”的工作流控制台

让 OpenAgents 不只是“能运行 workflow”，而是“能管理 workflow 的生命周期”。

---

## 三、总体节奏

| 迭代 | 主主题 | 核心目标 |
|------|------|------|
| Iteration 1 | 收口与加固 | 稳定契约、稳定状态、稳定主链路 |
| Iteration 2 | 恢复与观测 | 强化恢复能力、成本与质量观测 |
| Iteration 3 | 复盘与演进 | 强化版本差异、趋势分析、复盘体验 |

---

## 四、Iteration 1：收口与加固

### 主题

把现有能力做扎实。

### 目标

- 统一前后端契约
- 稳定 SSE 与执行状态一致性
- 收口运行主链路
- 建立关键自动化测试

### 重点任务

- DTO / API 契约收口
- SSE / Visual State 一致性加固
- Run 主链路闭环加固
- Diagnostics 初步深化
- Re-run / Comparison 基础增强
- 核心链路测试补强

### 交付结果

- Web 主链路稳定可用
- 执行页刷新和重连更可靠
- 关键链路具备自动化回归能力

### 本期价值

这一期是后三期的基础期。它的价值不在于新页面，而在于让系统进入“可持续迭代”状态。

---

## 五、Iteration 2：恢复与观测

### 主题

把系统从“能看问题”推进到“能恢复问题、能度量问题”。

### 目标

- 强化失败恢复能力
- 建立成本与质量观测能力
- 提升 workflow 运行效率与调试效率

### 建议重点内容

#### 1. 节点级恢复能力

- 从失败节点继续执行
- 节点级重试
- 基于依赖关系判断需要重跑的影响范围
- 对恢复动作给出风险提示

#### 2. 成本观测能力

- step / run 级 token 统计增强
- 成本估算面板
- 高耗时节点、高成本节点识别
- workflow 级运行成本摘要

#### 3. 质量观测能力

- eval 结果趋势展示
- 关键 workflow 的成功率/失败率
- gate 等待频次统计
- 失败类型分布

#### 4. 调试效率增强

- 基于历史运行的恢复建议
- rerun 影响范围提示
- compare 与 diagnostics 的联动增强

### 交付结果

- 用户不仅能看到失败，还能更快恢复
- 用户不仅能看到运行，还能理解成本与质量
- 系统开始具备“运维控制台”特征

### 本期价值

这一期会把 OpenAgents 从“可视化执行器”推向“可观测、可恢复的工作台”。

---

## 六、Iteration 3：复盘与演进

### 主题

把系统从“可恢复、可观测”推进到“可复盘、可持续优化”。

### 目标

- 强化 workflow 与 run 的版本/差异分析
- 建立趋势性复盘能力
- 让系统更适合持续迭代和团队使用

### 建议重点内容

结合当前实现节奏，Iteration 3 建议采用更聚焦的落地方式：

> 优先把“版本差异 + 历史趋势 + 失败复盘闭环”做透，而不是提前展开更重的平台化能力。

#### 1. 运行版本快照与差异视图

- 为 run 固化 workflow / config / agent / prompt 的版本上下文或快照引用
- 支持节点结构、模型、提示词、关键配置的差异对比
- 让版本变更与运行结果、恢复行为、质量波动建立可追踪关联

#### 2. 运行趋势分析

- 单个 workflow 的历史趋势
- 成功率、耗时、token、eval 趋势
- 异常波动识别
- 最近回归风险提示

#### 3. 复盘体验增强

- 失败 run 的结构化复盘摘要
- 关键节点变化、失败原因与恢复动作的关联说明
- compare、diagnostics、timeline 三者联动
- 更适合做 prompt / agent / workflow 调优决策

#### 4. 面向未来能力的结构预留

- 只做与版本快照、趋势指标、复盘聚合直接相关的结构预留
- 不提前展开团队协作、复杂资产管理或重报表系统
- 用轻量整理避免下一轮趋势与复盘功能返工

### 交付结果

- OpenAgents 不只是“执行一次工作流”，而是能帮助用户持续优化 workflow
- 系统开始具备“版本演进 + 结果复盘 + 趋势分析”的平台雏形

### 本期价值

这一期会让 OpenAgents 从“可靠控制台”进一步走向“智能化工作流运营台”。

---

## 七、三期依赖关系

三期之间建议按以下逻辑承接：

```text
Iteration 1
  稳定契约 / 状态 / 主链路 / 测试
      ↓
Iteration 2
  恢复能力 / 成本观测 / 质量观测
      ↓
Iteration 3
  版本差异 / 趋势分析 / 复盘增强
```

具体依赖关系：

1. 没有 Iteration 1 的契约和状态收口，Iteration 2 的恢复能力很容易做成脆弱补丁
2. 没有 Iteration 2 的恢复和观测基础，Iteration 3 的复盘和趋势分析很难有真实价值
3. 三期不是三组孤立功能，而是一条逐层增强的能力链

---

## 八、三期共同原则

### 原则 P1：主链路优先于外围能力

优先确保核心流程稳定，而不是先补边缘功能。

### 原则 P2：契约优先于页面

没有稳定数据模型时，页面做得越多，后续返工越大。

### 原则 P3：自动化回归优先于手工验证

未来三期都会持续调整状态模型和服务边界，测试基础必须同步增强。

### 原则 P4：做深优先于做广

每期尽量围绕 1-2 个核心主题做透，避免主题发散。

---

## 九、风险与控制建议

### 风险 1：过早继续加新模块

风险：

- 主链路未收口就继续扩功能
- 状态模型越来越分裂
- 页面越多，回归成本越高

建议：

- 每期先完成 P0 主线任务，再考虑新增展示型能力

### 风险 2：为了整理而做过重架构迁移

风险：

- 在主链路尚未稳定时大迁移，容易同时引入新问题

建议：

- 采用“伴随式整理”，优先做低风险结构优化

### 风险 3：测试滞后

风险：

- 功能做得越多，测试补得越晚，回归压力越大

建议：

- 每一期至少同步交付关键路径自动化测试

---

## 十、三期完成后的阶段判断

如果这三次迭代按上述路线完成，OpenAgents 将从当前状态进入一个新的阶段：

### 当前阶段

- 已具备 CLI + Web Workbench 的基础能力
- 能运行、能观察、能初步诊断

### 三期后目标阶段

- 主链路稳定
- 恢复能力成型
- 成本与质量可观测
- 版本与结果可复盘
- 能支撑后续更高阶产品化建设

届时再考虑更重的方向会更合适，例如：

- 更强的 workflow 版本管理
- 更完整的团队协作能力
- 更高级的可视化编辑能力

## 十一、建议与现有文档的关系

建议未来文档结构可逐步调整为：

- `docs/history/`
  - 历史 PRD
  - 历史任务拆解
  - 已完成阶段总结
- `docs/future/`
  - 下一期计划
  - 下一期任务拆解
  - 三期路线图

这样可以把“已经发生的设计决策”和“未来规划”明确分开，方便持续维护。
