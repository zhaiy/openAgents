# OpenAgents 下一期质检与审查清单

> 创建日期：2026-03-24
> 适用范围：下一期迭代开发完成后的质检、代码审查与回归验收
> 目标：用统一清单审查模型产出是否真实可用

---

## 一、使用方式

后续无论由谁开发，质检和审查建议按以下顺序执行：

1. 先审契约
2. 再审主链路
3. 再审异常场景
4. 再审测试覆盖
5. 最后审是否引入额外复杂度

---

## 二、契约审查

需要检查：

- `src/app/dto.ts` 是否仍是核心 DTO 真源
- route 返回结构是否与 DTO 一致
- `web/src/api/index.ts` 是否只是消费契约，而不是重新定义契约
- 页面是否依赖隐式兼容字段
- 时间字段是否统一
- 错误响应结构是否统一

高风险信号：

- 同一实体出现两套命名
- `id`、`runId`、`workflowId` 等混用
- 页面出现大量 `foo ?? bar ?? baz` 的兼容判断
- 通过文案文本而非错误码驱动逻辑

---

## 三、状态一致性审查

需要检查：

- visual state 是否有单一真源
- `run-store` 与 `graph-store` 是否职责清晰
- snapshot 与增量事件边界是否明确
- sequence / reconnect 逻辑是否完整
- 图节点、Inspector、timeline 是否一致

高风险信号：

- 同一状态在多个 store 双写
- 前端用大量兜底逻辑掩盖事件顺序问题
- 刷新后恢复依赖偶然的页面初始化分支

---

## 四、主链路审查

需要检查：

- 启动运行的输入校验是否可解释
- Gate waiting 和 gate resolved 反馈是否一致
- rerun / rerun-with-edits 行为边界是否清楚
- Runs、Run Detail、Run Execution 页面是否状态衔接自然
- run / step / draft / compare session 不存在时反馈是否统一

高风险信号：

- 正常流能跑通，但异常流全靠页面特判
- rerun 实际依赖不透明的历史状态
- 页面跳转后状态脱节

---

## 五、Diagnostics / Compare / Recovery 审查

需要检查：

- Diagnostics 是否能说明失败点、影响面和建议动作
- Compare 是否能支持判断，而不仅是并排展示
- Recovery 相关数据模型是否清楚
- 本期是否只做了“显示更多字段”，却没有增强可操作性

高风险信号：

- diagnostics 字段来源不可解释
- compare 页面字段变多，但没有摘要能力
- recovery 预留接口含义模糊

---

## 六、测试与回归审查

需要检查：

- 是否补了路由测试
- 是否补了主链路 integration 测试
- 是否补了 smoke / E2E
- 测试是否覆盖刷新恢复与断线重连
- 测试是否覆盖对象不存在或 session 失效

最低通过建议：

- 至少一条完整主链路自动化覆盖
- 至少一组异常场景自动化覆盖

---

## 七、复杂度控制审查

需要检查：

- 是否顺手做了大重构
- 是否新增了多余抽象层
- 是否引入了与当前风格不一致的新模式
- 是否把本应统一的问题做成局部 patch

高风险信号：

- 在 `N1-N3` 中顺带重写路由组织方式
- 为兼容旧逻辑保留大量临时分支
- 页面和 service 同时开始自定义转换层

---

## 八、模型适配与升级建议

## 1. 适合继续使用 `gpt-5.3-codex`

满足以下条件时优先使用：

- 任务涉及系统边界定义
- 涉及跨层契约、状态真源或主链路语义
- 需要先做“定边界 + 最小实现 + 测试保护”

## 2. 适合继续使用 `GLM-5`

满足以下条件时可继续：

- 任务边界已经写清
- 改动主要是 service、DTO、路由、集成测试或对比/诊断增强
- 需要中高复杂度工程实现，但不需要重新定义系统级边界

## 3. 适合继续使用 `Kimi K2.5`

满足以下条件时可继续：

- 任务边界已经写清
- 改动主要在页面、文案、异常态、可视化表现、基础测试
- 不需要重新定义契约或状态真源

## 4. 适合继续使用 `qwen3.5`

满足以下条件时可继续：

- 任务边界已经写清
- 改动主要在页面、文案、异常态、基础测试
- 不需要重新定义契约或状态真源

## 5. 适合继续使用 `minimax m2.7`

满足以下条件时可继续：

- 任务是明确边界下的 service / DTO / compare / diagnostics 增强
- 涉及跨层修改，但契约真源已明确
- 有清晰测试要求

## 6. 建议更换为更强模型的情形

出现以下情况时，建议停止让 `GLM-5`、`Kimi K2.5`、`qwen3.5` 或 `minimax m2.7` 继续主导：

1. 连续两轮修改后，契约仍反复漂移
2. SSE 重连、刷新恢复、多 store 状态一致性始终不稳
3. 模型开始通过新增兼容字段绕过根因
4. 主链路行为边界无法稳定收口
5. 需要决定 snapshot、event patch、rerun 语义等系统级边界
6. 代码改动明显超过任务卡允许范围

建议升级方向：

- 优先换到 `gpt-5.3-codex` 或同级更强的通用编码模型
- 先让强模型定边界，再交回中等模型继续落地

---

## 九、质检结论模板

后续审查时，建议统一输出以下结构：

1. 本次是否通过
2. 主要发现
3. 高风险问题
4. 建议返工项
5. 是否需要更换模型
6. 回归验证结果

---

## 十、最终判断标准

只有当以下条件同时满足，才建议判定“本任务可合入或可进入下一步”：

1. 契约未继续漂移
2. 主链路可解释且可回归
3. 异常流不是靠页面 patch 勉强维持
4. 测试能发现关键回归
5. 复杂度没有明显失控
