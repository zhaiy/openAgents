# OpenAgents 模型开发分配指南

> 创建日期：2026-03-24
> 适用范围：`docs/future/` 下的下一期迭代任务分发
> 目标：让不同大模型以更低风险介入开发

---

## 一、文档目的

本文档用于回答三个问题：

1. 哪些任务适合直接交给模型开发
2. `gpt-5.3-codex`、`GLM-5`、`Kimi K2.5`、`qwen3.5`、`minimax m2.7` 分别更适合做什么
3. 当模型能力不够时，何时应升级到更强模型

---

## 二、总体原则

### 原则 P1：先给边界，再给实现

模型最容易做错的，不是“不会写代码”，而是“边界不清时做了错误决策”。

因此，分发任务时必须同时提供：

- 对应任务卡
- 契约真源文档
- 场景矩阵
- 验收口径

### 原则 P2：高风险任务不要只给一句自然语言

以下任务不应只给一句“帮我实现一下”：

- DTO 契约收口
- SSE 一致性
- store 职责收敛
- rerun / recovery 行为定义

这些任务都必须附上明确的非目标、修改范围和验收标准。

### 原则 P3：测试与实现同步分发

不要让模型先改完功能、最后再补测试。

建议至少采用以下节奏：

1. 先改契约或状态模型
2. 同步补 route / service / integration 测试
3. 再做页面收口
4. 最后做 smoke / E2E 验证

### 原则 P4：模型不负责决定产品边界

模型可以实现、整理、补测试、收口异常处理，但不应擅自扩展产品范围。

如果任务中出现以下情况，应暂停并升级决策：

- 需要新增一个此前未约定的新页面或新主流程
- 需要改变主链路行为语义
- 需要引入较重框架迁移
- 需要修改多个任务包之间的依赖关系

## 三、模型能力分配建议

## 1. `gpt-5.3-codex` 适合承担的任务

更适合以下类型：

- 系统级边界定义
- 跨层契约收口
- SSE / store / route / service 协同修改
- 主链路行为收口
- 低风险结构整理与测试同步补齐

适合优先承担的任务：

- `N1` DTO / API 契约收口
- `N2` SSE 与执行状态一致性加固
- `N3` Run 主链路闭环加固
- `N8` 路由层与服务层整理

建议角色：

- 先定边界
- 再做最小必要实现
- 最后给中等模型留出可继续接手的稳定结构

## 2. `GLM-5` 适合承担的任务

更适合以下类型：

- 明确边界下的中高复杂度工程任务
- service 层增强
- DTO / route / 前端类型的跨层同步修改
- diagnostics / compare / recovery 相关能力增强
- integration / route 测试补强

适合优先尝试的任务：

- `N4` 核心链路测试补强
- `N5` Diagnostics 深化
- `N6` Re-run / Recovery 增强
- `N7` Comparison 深化
- 在边界已明确前提下参与 `N1`、`N3` 落地

不建议在边界不清时独立主导：

- `N2` 的事件一致性核心机制
- `N8` 的结构性整理

## 3. `Kimi K2.5` 适合承担的任务

更适合以下类型：

- 页面收口
- 前端交互与展示层增强
- 可视化与对比界面体验增强
- 文案、空态、异常态收口
- smoke / E2E / 页面行为测试补充

适合优先尝试的任务：

- `N3` 中页面反馈和导航衔接子任务
- `N4` 中 smoke / E2E 子任务
- `N5`、`N7` 中页面层增强部分

不建议直接独立承担：

- `N1` 的整体契约定稿
- `N2` 的 SSE 一致性主设计
- 涉及多个 store 职责重排的任务

## 4. `qwen3.5` 适合承担的任务

更适合以下类型：

- 标准页面收口
- DTO 字段对齐后的前端消费调整
- 文案、错误提示、空态和异常态收口
- 低到中复杂度测试补充
- 已有模式下的 API / 页面增强

适合优先尝试的任务：

- `N4` 核心链路测试补强中的页面 smoke / E2E 子任务
- `N7` Comparison 深化中的展示层增强部分
- `N3` 主链路中的异常提示统一、导航状态衔接

不建议直接独立承担：

- `N1` 的整体契约定稿
- `N2` 的 SSE 一致性主设计
- 涉及多个 store 职责重构的任务

## 5. `minimax m2.7` 适合承担的任务

更适合以下类型：

- service 层增强
- DTO / 路由 / 前端类型的跨层同步修改
- diagnostics / compare 的结构化数据增强
- 中复杂度的测试补强

适合优先尝试的任务：

- `N1` 中字段和错误结构的统一落地
- `N5` Diagnostics 深化
- `N6` Re-run / Recovery 的数据模型收口
- `N7` Comparison DTO 与 service 增强

不建议在缺少明确边界时独立承担：

- `N2` 的事件重连和状态恢复核心机制设计
- 需要重排多个页面与 store 职责的任务

## 四、推荐分配策略

### Phase A：P0 主线阶段

#### A1. 高能力模型先定边界

优先由更强模型或人工先完成：

- `N1` 的 DTO 真源与错误模型定义
- `N2` 的 snapshot / event patch / reconnect 语义定义
- `N3` 的主链路行为边界定义

本阶段优先级建议：

1. `gpt-5.3-codex`
2. 具备稳定工程能力的更强通用编码模型
3. 其他模型仅在边界明确后接手落地

#### A2. 中等模型再落实现

在边界稳定后，可将以下子任务分发给 `GLM-5`、`Kimi K2.5`、`qwen3.5` 或 `minimax m2.7`：

- 前端 API 类型同步
- 页面异常态和提示收口
- route / service 测试补充
- smoke / E2E 回归补充

### Phase B：价值增强阶段

这一阶段更适合并行：

- `GLM-5` 或 `minimax m2.7` 负责 diagnostics / recovery / compare 的 service 和 DTO
- `Kimi K2.5` 或 `qwen3.5` 负责对应页面展示和交互收口

### Phase C：适度整理阶段

`N8` 不建议优先交给中等能力模型独立主导。

如必须由模型参与，应限定为：

- 路由辅助函数提取
- 明显重复的错误处理抽取
- 不改变外部行为的低风险整理

## 五、建议升级模型的信号

如果出现以下任一情况，建议暂停当前模型并更换为更强模型：

1. 连续两次提交都在修补同一类契约漂移问题
2. 修改后能跑通局部测试，但主链路行为语义明显偏离任务卡
3. 开始通过增加兼容分支来“绕过”不理解的数据模型
4. 在 store、SSE、route、service 之间频繁出现重复字段映射
5. 需要同时修改后端事件模型、前端状态模型和页面行为
6. 提交中出现较大范围“顺手重构”，但没有对应测试保护

## 六、任务与模型建议映射

| 任务 | 建议主力 | 建议配合 | 说明 |
|------|------|------|------|
| `N1` DTO / API 契约收口 | `gpt-5.3-codex` | `GLM-5` | 先定真源，再分发类型同步和测试 |
| `N2` SSE / 状态一致性 | `gpt-5.3-codex` | `GLM-5` | 高风险状态任务，不建议直接交中等模型独立主导 |
| `N3` 主链路闭环加固 | `gpt-5.3-codex` 或人工定边界 | `GLM-5` / `Kimi K2.5` | 可拆成异常态、导航、反馈、测试子任务 |
| `N4` 核心链路测试补强 | `GLM-5` | `Kimi K2.5` / `qwen3.5` | 适合并行分发 |
| `N5` Diagnostics 深化 | `GLM-5` | `Kimi K2.5` | service + page 分拆效果较好 |
| `N6` Re-run / Recovery 增强 | `gpt-5.3-codex` 或 `GLM-5` | `Kimi K2.5` | 先定模型关系，再做页面 |
| `N7` Comparison 深化 | `GLM-5` | `Kimi K2.5` | 较适合结构化增强 |
| `N8` 路由层整理 | 更强模型 | 无或少量配合 | 容易失控，不建议轻率下放 |

## 七、建议执行方式

建议实际分发时使用以下格式：

1. 指定任务编号，例如 `N3`
2. 附上对应任务卡
3. 附上相关契约真源或场景矩阵
4. 明确允许修改的文件范围
5. 明确必须补充的测试
6. 明确禁止事项
   - 不新增未约定页面
   - 不做大重构
   - 不引入兼容分支掩盖契约问题
7. 要求输出：
   - 修改摘要
   - 风险说明
   - 测试结果
   - 未完成项

---

## 八、最终建议

如果你接下来主要使用 `gpt-5.3-codex`、`GLM-5`、`Kimi K2.5`，建议采用以下策略：

1. `N1/N2/N3/N8` 优先交给 `gpt-5.3-codex` 或同级更强模型
2. `N4/N5/N6/N7` 可以较早安排 `GLM-5` 介入
3. `Kimi K2.5` 更适合承担页面收口、异常态收口和部分 smoke / E2E 任务
4. 一旦出现契约反复漂移、SSE 状态反复不稳、主链路行为定义不清，应立即更换到更强模型，不要继续堆补丁
