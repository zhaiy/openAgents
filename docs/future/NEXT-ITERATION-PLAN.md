# OpenAgents 下一期迭代计划

> 创建日期：2026-03-27
> 适用阶段：作为当前项目的最后一次产品化收尾迭代
> 规划主题：产品化收口与标准化 Skill 收尾

---

## 一、迭代定位

经过前几轮迭代，OpenAgents 已经具备较完整的工作流编排、恢复、观测、复盘能力。  
从当前项目目标来看，OpenAgents 的核心定位已经越来越清晰：

> **它不是再做一个新的 Agent 产品，而是作为多种 Agent 工具之上的工作流编排层与调度底座。**

因此，最后一轮迭代不建议继续横向扩功能，不建议再投入新的页面簇，也不建议为了“支持更多 Agent 工具”去做一套重型 Adapter 体系。

这轮更适合做的，是把 OpenAgents 收口成一个：

1. 边界清晰的编排产品
2. 对外有统一接入约定的调度底座
3. 对内安全、稳定、易复用的工程基础

一句话概括：

> **最后一轮不是“再做更多”，而是把 OpenAgents 从“功能已经够用的系统”，收成“适合长期复用的 Agent Orchestration Core”。**

---

## 二、关键判断

### 1. 不做 Adapter 优先，先做标准化 Skill 规范

考虑到目前主流 Agent 工具已经普遍支持 skill 概念，且它们具备读取文档、理解 CLI、自己完成工具接入的能力，
本项目最后一轮不优先投入“为每家 Agent 手工做 Adapter”。

更值得做的是：

- 提供一套清晰、稳定、文档化的 Skill 规范
- 定义 skill 如何声明能力、输入、输出、依赖、调用方式与安全边界
- 让未来不同厂商的 Agent/LLM 可以参考这套规范，自主完成对 OpenAgents 能力的理解与调用

换句话说：

> **OpenAgents 更应该输出“接入标准”，而不是继续堆“适配代码”。**

### 2. 不补模板库，保留谨慎策略

本轮不建议为了完整性去提前补大量模板和示例工程。

原因是：

- 模板一旦进入仓库，就默认代表项目推荐实践
- 如果模板未经你真实使用和评判，后续反而会误导用户
- 对当前收尾目标而言，模板库不是主矛盾

因此本轮原则是：

> **只完善接入规范、操作说明和最小可验证路径，不扩模板库。**

---

## 三、本轮目标

### 目标 G1：定义标准化 Skill 规范

围绕 OpenAgents 的真实定位，形成一套可公开说明、可被不同 Agent 工具参考的 Skill 规范。

重点包括：

- skill 元数据
- 能力声明
- 输入输出约定
- CLI 调用约定
- 错误与退出码约定
- 安全边界说明

### 目标 G2：建立启动前健康检查与运行前诊断

让用户在真正运行 workflow 前，就知道当前环境是否“可运行”。

重点包括：

- 配置完整性检查
- API key / runtime / workflow / agent / skill 可用性检查
- 风险项提示
- 可执行修复建议

### 目标 G3：收口高风险执行能力的安全边界

把脚本执行、post-processor、webhook、外部命令等高风险能力的默认策略和边界明确下来，并统一到文档和校验中。

重点包括：

- 默认安全策略
- 受信任本地执行边界
- 可配置放开项
- 面向用户的风险提示

### 目标 G4：稳定对外运行契约与可观测字段

既然 OpenAgents 要作为调度底座，最后一轮应保证它对外暴露的运行契约足够稳定。

重点包括：

- CLI 主命令口径
- 运行状态/事件字段口径
- 失败分类口径
- 输出目录与产物约定

### 目标 G5：完成发布收尾

让项目在最后一轮结束后具备“可交付、可说明、可维护”的状态。

重点包括：

- README 和能力边界说明
- 接入说明文档
- 版本说明 / 发布前检查清单
- 回归测试基线

---

## 四、本轮原则

1. 不新增大功能模块，不重做 UI 结构。
2. 不做多厂商 Adapter 体系，优先输出 Skill 标准。
3. 不为了完整性补模板库，模板继续坚持“真实用过再收录”。
4. 先做边界、契约、安全，再做包装性文档。
5. 每个任务包都要带测试或验证方式，避免只补文档不补落地。
6. 优先改善“每月做新产品时会重复踩坑”的部分。

## 六、正式任务包

## F1. 标准化 Skill 规范

### 目标

定义一套可被 Agent 工具和 LLM 理解、参考、接入的 Skill 规范，作为 OpenAgents 的长期接入标准。

### 重点内容

- Skill 文件结构与最小字段集
- 能力声明与输入输出格式
- CLI 调用建议与退出码约定
- 依赖、权限、风险标签约定
- Skill 文档编写模板

### 交付物

- Skill 规范文档
- Skill 示例骨架
- Skill 规范校验建议

### 优先级

`P0`

---

## F2. 健康检查与运行前诊断

### 目标

在 workflow 运行前提供一套“预检能力”，帮助用户提前发现环境问题、配置问题和明显风险。

### 重点内容

- 新增 preflight / doctor 类命令或等价能力
- 检查 runtime 默认配置和密钥可用性
- 检查 workflow / agent / skill 引用完整性
- 检查高风险配置项
- 输出结构化诊断结果

### 交付物

- 运行前诊断命令
- 诊断结果结构
- 相关测试

### 优先级

`P0`

---

## F3. 安全边界与执行策略收口

### 目标

明确 OpenAgents 的执行安全边界，把高风险能力的默认策略和可放开项收口成一套稳定规则。

### 重点内容

- script runtime / post-processor / webhook 安全边界说明
- 配置校验与默认策略补强
- 风险级别标记
- README 与文档中的安全说明

### 交付物

- 安全边界说明文档
- 相关校验或默认策略增强
- 相关测试

### 优先级

`P0`

---

## F4. 运行契约与事件口径收口

### 目标

把 OpenAgents 作为调度底座对外暴露的关键契约收口清楚，避免后续复用时发生口径漂移。

### 重点内容

- CLI 命令和参数口径
- run / step / event 字段语义梳理
- 输出目录与产物约定
- 错误分类和退出码说明

### 交付物

- 运行契约说明文档
- 必要的字段整理
- 契约测试

### 优先级

`P1`

---

## F5. 文档与发布收尾

### 目标

完成最后一轮面向用户与后续维护者的文档整理，让项目进入可发布、可交接状态。

### 重点内容

- README 收口
- Quick Start 路径校验
- 接入说明和能力边界说明
- 发布前检查清单
- 版本发布说明草案

### 交付物

- 更新后的 README / docs
- 发布检查清单
- 发布说明草案

### 优先级

`P0`

---

## F6. 回归测试与收尾验证

### 目标

把本轮新增的标准化规范、健康检查、安全边界、契约口径纳入自动化测试和手动验收路径。

### 重点内容

- preflight / doctor 测试
- Skill 规范相关校验测试
- 安全边界测试
- README / Quick Start 手动验收清单

### 交付物

- 测试补强
- 验收清单
- 收尾验证结果

### 优先级

`P0`

---

## 七、不在本轮范围内

- 不做新的大页面或新的控制台模块
- 不做多厂商专用 Adapter 体系
- 不扩充模板库和示例库
- 不把项目扩展成团队协作平台
- 不做复杂部署平台或 SaaS 化改造

---

## 八、建议完成顺序

建议按以下顺序推进：

1. `F1` 标准化 Skill 规范
2. `F3` 安全边界与执行策略收口
3. `F2` 健康检查与运行前诊断
4. `F4` 运行契约与事件口径收口
5. `F5` 文档与发布收尾
6. `F6` 回归测试与收尾验证

关键路径如下：

```text
F1 -> F3 -> F2 -> F4 -> F5 -> F6
```

---

## 九、验收标准

- 项目对外定位清晰，不再模糊为“又一个 Agent 产品”
- Skill 规范可独立阅读和实现，不依赖口头解释
- 用户在运行前能发现主要环境问题
- 高风险能力的边界、默认策略、放开方式都已明确
- CLI / run / event / output 的关键口径稳定
- README 能带用户完成首次可验证路径
- 本轮改动具备自动化回归保护
