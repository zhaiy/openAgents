# OpenAgents 下一期迭代计划

> 创建日期：2026-03-22
> 适用阶段：v0.5.0 之后的下一期迭代
> 规划主题：功能收口、架构加固、核心流程做扎实

---

## 一、迭代定位

当前版本已经基本完成从 CLI 工作流引擎到本地优先 Web Workbench 的第一轮升级，前后端已具备以下核心能力：

- Workflow Overview
- Visual Run Config
- Execution Console
- Diagnostics
- Run Comparison Lite
- Re-run / Edit-and-Re-run
- Home / Runs / Settings 升级

下一期迭代不再以横向扩展新功能为主，而是转向以下目标：

1. 把现有能力做稳，减少前后端契约漂移和状态不一致问题
2. 把核心链路做深，让“运行、观察、诊断、恢复”形成更强闭环
3. 把流程引擎和可视化控制台之间的数据模型沉淀下来，为后续两到三期持续打基础

一句话概括：

> 下一期不是“继续堆功能”，而是“让 OpenAgents 现有控制台和引擎真正进入可持续演进状态”。

---

## 二、迭代目标

### 目标 G1：稳定前后端数据契约与状态一致性

解决 Run、Visual State、Timeline、SSE、Summary DTO 在不同层之间字段语义不完全统一的问题，降低联调和回归成本。

### 目标 G2：强化核心运行闭环

围绕 `启动运行 -> 实时观察 -> Gate 处理 -> 失败诊断 -> 重跑/恢复` 建立更可靠、更容易解释的主链路。

### 目标 G3：提升工程质量与可回归性

让当前能力不只“能演示”，而是具备稳定开发、稳定联调、稳定回归的工程基础。

### 目标 G4：为后续两期持续夯实基础

提前为“节点级恢复”“更深的 compare”“成本与质量观测”等能力预留结构，不在后续迭代里反复返工。

---

## 三、本期重点原则

1. 不以新增页面数量作为主要目标。
2. 优先处理跨层契约、核心链路、状态一致性问题。
3. 每个任务包必须尽量形成闭环，避免只做一半 API 或一半 UI。
4. 每个高优先级任务都应附带测试或验收方式。
5. 不主动引入高风险大迁移，除非它能明显降低后续 2-3 期成本。

## 五、正式任务包

## F1. 前后端 DTO 与 API 契约收口

### 目标

统一 `src/app/dto.ts`、`src/web/routes.ts`、`web/src/api/index.ts`、各页面真实使用字段之间的语义与命名，减少“后端返回 A，前端期待 B”的隐性问题。

### 重点内容

- 统一 Run Summary / Run Detail / Visual State / Timeline 的字段命名与类型
- 统一时间字段格式策略
  - 明确哪些接口返回 `timestamp number`
  - 明确哪些接口返回 `ISO string`
- 统一错误响应结构
- 对现有 API 做最小必要的 schema 校验
- 减少前端手写兼容逻辑和隐式字段推断

### 交付物

- 一版稳定的 Web API DTO 契约
- API 错误模型说明
- 关键接口的契约测试或路由测试

### 优先级

`P0`

### 预计收益

- 前后端联调成本明显下降
- 页面升级时更不容易出现静默错误
- 为后续 diagnostics / compare / recovery 提供统一基础

---

## F2. SSE / Visual State 一致性加固

### 目标

让执行过程中的实时状态更新具备更高一致性，确保首次进入、断线重连、连续事件、页面刷新后都能正确恢复。

### 重点内容

- 明确 snapshot 与增量事件的边界
- 固化事件序号、last event id、重连恢复逻辑
- 统一 run-store 与 graph-store 的状态来源
- 避免 React Flow 节点状态与 Inspector 状态“双轨漂移”
- 补充 `step.started / stream / completed / failed / gate.waiting / gate.resolved / workflow.completed` 的完整更新策略

### 交付物

- SSE 事件一致性策略文档
- run execution 页面状态合并逻辑收口
- 重连与刷新恢复测试

### 优先级

`P0`

### 预计收益

- 执行控制台更可靠
- 降低“节点状态显示不准”“刷新后状态丢失”的问题
- 让后续可视化能力建立在稳定事件流之上

---

## F3. Run 核心主链路闭环加固

### 目标

把最关键的业务主链路打磨成“真实可用”的核心闭环，而不只是页面存在。

### 重点内容

- 启动运行时的输入校验、错误提示和预运行摘要收口
- Gate 等待、处理、恢复的交互一致性
- rerun / rerun-with-edits 的行为定义收口
- run 列表、详情、执行页之间的导航与状态联动收口
- 异常场景下的用户反馈统一
  - run 不存在
  - step 不存在
  - draft 不存在
  - compare session 失效

### 交付物

- 运行主链路验收清单
- 关键异常场景处理补齐
- 对应的 smoke / integration 测试补齐

### 优先级

`P0`

### 预计收益

- 用户可以更稳定地完成一次完整流程
- 真实使用中的阻塞点会明显减少
- 后续再扩展能力时，不会频繁打断主链路

---

## F4. Diagnostics 深化

### 目标

让 Diagnostics 从当前的“失败摘要与建议”升级为“更有定位能力的诊断面板”。

### 重点内容

- 明确 failed node、gate waiting node、upstream state、downstream impact 的真实语义
- 补齐 downstream impact 分析
- 增加失败传播解释
- 增加更细粒度的错误分类
- 给出更可操作的恢复建议
- 与 execution console 联动，支持从诊断跳转到具体节点上下文

### 交付物

- 更完整的 `DiagnosticsSummaryDto`
- Diagnostics 页面增强版
- 失败链路分析测试

### 优先级

`P1`

### 预计收益

- 用户更快定位失败原因
- 降低“看见失败但不知道怎么办”的情况
- 为后续恢复能力提供依据

---

## F5. Re-run / Recovery 能力增强

### 目标

在现有“整次 rerun”基础上，进一步提升恢复效率和复用能力。

### 重点内容

- 历史输入复用体验优化
- rerun 前展示输入差异与风险提示
- 为后续“节点级恢复 / 从失败点继续”预留服务接口
- 明确 config draft、historical config、rerun payload 三者的关系
- 减少重复运行时的无效操作

### 交付物

- Re-run 体验增强版
- 更清晰的 reusable config 模型
- recovery 预留接口设计说明

### 优先级

`P1`

### 预计收益

- 重跑效率更高
- 更适合真实调试和参数迭代场景
- 为后两期恢复类能力铺路

---

## F6. Comparison 能力深化

### 目标

把当前轻量 compare 从“基础对比”升级为“有决策价值的复盘工具”。

### 重点内容

- 比较结果不仅包含 status / duration / token
- 增加关键节点差异摘要
- 增加输入差异可读化展示
- 对输出差异给出结构化 summary，而不只是计数
- 明确 compare session 的生命周期与定位

### 交付物

- 更完整的 Comparison DTO
- Run Comparison 页面增强版
- compare service 测试补齐

### 优先级

`P1`

### 预计收益

- 更适合做 prompt / 参数 / workflow 迭代验证
- 降低人工比对成本
- 为后续质量趋势分析打基础

---

## F7. 测试与质量体系补强

### 目标

从当前以 service 单测为主，升级到“核心链路可回归”的质量体系。

### 重点内容

- 保持现有单元测试优势
- 增加 API 路由测试覆盖
- 增加围绕运行主链路的集成测试
- 增加 Web 端关键页面的端到端测试
- 覆盖关键链路：
  - 创建运行
  - SSE 执行更新
  - Gate 处理
  - 诊断查看
  - compare
  - rerun

### 交付物

- 测试矩阵
- 一组能覆盖主链路的 E2E / integration 用例
- CI 可稳定执行的最小回归集合

### 优先级

`P0`

### 预计收益

- 降低后续 3 次迭代的回归风险
- 提升重构和收口工作的信心
- 让“扎实”真正可验证

---

## F8. 路由层与服务层适度整理

### 目标

控制当前路由和服务增长带来的复杂度，但避免在本期做过重迁移。

### 重点内容

- 拆分 `src/web/routes.ts` 的职责
- 引入统一的路由注册、参数解析、错误处理帮助函数
- 评估是否在下一期之后引入更成熟的轻量框架
- 清理明显重复的服务逻辑和转换逻辑

### 交付物

- 路由层整理方案
- 若干低风险重构提交
- 技术债记录

### 优先级

`P2`

### 预计收益

- 避免后续 API 继续膨胀成单文件瓶颈
- 提高可维护性
- 为再下一期的观测/恢复能力预留空间

---

## 六、建议开发顺序

建议按以下顺序推进：

1. `F1 DTO / API 契约收口`
2. `F2 SSE / Visual State 一致性加固`
3. `F3 Run 核心主链路闭环加固`
4. `F7 测试与质量体系补强`
5. `F4 Diagnostics 深化`
6. `F5 Re-run / Recovery 能力增强`
7. `F6 Comparison 能力深化`
8. `F8 路由层与服务层适度整理`

原因：

- `F1 + F2 + F3 + F7` 共同构成“把当前系统做扎实”的最小核心闭环
- `F4 + F5 + F6` 属于在稳定基础上的价值增强
- `F8` 重要，但更适合作为伴随式整理，而不是抢占核心链路资源

## 八、预计收益清单

### 产品层收益

- 现有控制台从“功能可见”升级为“能力可靠”
- 用户能更顺畅完成运行、观察、诊断、恢复闭环
- 为后续 2-3 期持续优化奠定稳定基础

### 工程层收益

- 联调成本下降
- 回归风险下降
- 重构空间变大
- 新功能接入成本下降

### 架构层收益

- 引擎与 Web 控制台的数据边界更清楚
- 事件流模型更稳定
- 后续恢复、成本、质量分析能力更容易接入

---

## 九、验收标准

下一期迭代完成后，至少应满足以下验收结果：

1. Web 主链路可以稳定完成：
   - 配置运行
   - 实时观察
   - Gate 处理
   - 失败诊断
   - rerun / edit-and-rerun
2. 执行页在刷新和重连后，状态恢复可预期。
3. 前后端核心 DTO 的字段语义清晰且基本统一。
4. Diagnostics 与 Comparison 不再只是“展示页”，而具备实际操作参考价值。
5. 自动化测试可以覆盖至少一条完整核心链路。

---

## 十、对后续两期的衔接建议

本期完成后，后续两期建议继续沿着“做深而不是做散”的方向推进：

- 迭代二：节点级恢复、失败点继续执行、成本与质量观测增强
- 迭代三：Workflow 版本差异、质量趋势分析、恢复与复盘体验进一步产品化

这意味着本期最重要的任务，不是追求更多页面，而是确保现有控制台、服务层、状态模型和流程引擎具备持续演进能力。
