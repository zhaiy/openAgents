# OpenAgents 下一期迭代计划

> 创建日期：2026-03-24
> 适用阶段：`v0.5.0` 收口迭代完成之后的下一期
> 规划主题：恢复与观测

---

## 一、迭代定位

上一期已经重点完成以下方向：

- 前后端契约收口
- 执行状态一致性加固
- Run 主链路闭环增强
- Diagnostics / Re-run / Comparison 基础增强
- 核心测试补强

下一期不再继续以“接口收口”和“主链路补漏”为主，而是进入第二层能力建设：

1. 让失败后的处理从“重新跑一次”升级到“更精细地恢复”
2. 让运行结果从“看状态”升级到“看成本、看质量、看风险”
3. 为后续的趋势分析、版本差异和复盘体验建立稳定统计口径

一句话概括：

> 下一期不是继续修边角，而是把 OpenAgents 从“稳定控制台”推进到“可恢复、可观测的工作台”。

---

## 二、迭代目标

### 目标 G1：建立可解释的恢复能力

围绕失败节点、依赖关系和历史运行结果，建立“恢复前可预览、恢复时可控制、恢复后可验证”的能力闭环。

### 目标 G2：建立成本观测能力

让用户不仅知道 run 是否成功，还能知道耗时和 token 主要花在什么地方，为 prompt、agent、workflow 调优提供依据。

### 目标 G3：建立质量观测能力

沉淀 workflow 级质量指标，让成功率、失败率、gate 等待频次、失败类型分布、eval 摘要等指标可被稳定观察。

### 目标 G4：继续加固工程回归基础

恢复和观测都会引入新的服务层逻辑与统计口径，必须同步用自动化测试保护，否则后续趋势分析会建立在脆弱基础上。

---

## 三、本期重点原则

1. 恢复能力优先于新展示页面。
2. 先定义恢复语义和统计口径，再做页面表达。
3. 每个任务包必须包含 API、页面或测试中的至少两层闭环。
4. 不用展示层特判去掩盖恢复语义不清或统计口径不稳。
5. 不做大规模重构，优先做可验证、可逐步演进的增强。

## 五、正式任务包

## R1. 节点级恢复能力

### 目标

把当前“整次 rerun”为主的能力推进到“从失败节点继续执行 / 节点级重试 / 基于依赖关系判断影响范围”的恢复能力。

### 重点内容

- 明确“从节点恢复”和“整次 rerun”的行为边界
- 支持从失败节点或指定节点发起恢复
- 基于依赖关系判断需要重跑、可复用、必须失效的节点范围
- 收口恢复动作和原 run / source run 的关联关系
- 给恢复后的 run 提供明确标识与可追踪信息

### 交付物

- 一版稳定的 recovery 语义定义
- 恢复相关 DTO / API / service 实现
- 恢复路径测试

### 优先级

`P0`

### 预计收益

- 失败后不必总是整次重跑
- 用户可以更快进入“修复 -> 验证”闭环
- 为后续更深的恢复建议提供基础

---

## R2. 恢复预览与风险提示

### 目标

让恢复动作在执行前具备可预览、可解释、可提示风险的能力，而不是盲目触发。

### 重点内容

- 在恢复前展示受影响节点范围
- 区分“将复用”“将重跑”“结果将失效”的节点集合
- 对 gate、eval、下游输出变化风险给出提示
- 给 rerun / rerun-with-edits / recover 形成统一 preview 体验
- 在 diagnostics、comparison、execution console 之间补充恢复入口联动

### 交付物

- Recovery Preview DTO
- 恢复风险提示说明
- 页面侧恢复预览面板

### 优先级

`P0`

### 预计收益

- 恢复动作更可控、更容易解释
- 减少“恢复后结果为什么变化了”的困惑
- 让 diagnostics 真正形成行动闭环

---

## R3. 成本观测能力

### 目标

把 token 与耗时信息从“字段存在”升级为“可用于判断和调优的观测能力”。

### 重点内容

- step / run 级 tokenUsage 汇总增强
- 耗时与 token 的结构化聚合
- 高耗时节点、高 token 节点识别
- run 详情与 comparison 中补充成本视角
- workflow 级成本摘要基础接口

### 交付物

- 成本观测 DTO 与聚合服务
- 成本面板或摘要卡片
- 成本统计测试

### 优先级

`P1`

### 预计收益

- 更容易定位“哪里慢”“哪里贵”
- 更适合做 prompt / model / workflow 调优
- 为后续趋势分析提供统一成本口径

---

## R4. 质量观测能力

### 目标

建立 workflow 级质量指标，让系统开始具备质量运营视角。

### 重点内容

- workflow 级成功率 / 失败率摘要
- gate 等待频次统计
- 失败类型分布统计
- 最近运行的 eval 摘要聚合
- 基础异常波动提示预留字段

### 交付物

- 质量观测 DTO 与服务层聚合
- 质量概览页面或首页摘要增强
- 质量统计测试

### 优先级

`P1`

### 预计收益

- 用户可以更快判断某个 workflow 是否健康
- diagnostics / comparison 不再只是单次 run 视角
- 为第三期趋势分析和复盘增强提供真实数据基础

---

## R5. 恢复与观测测试补强

### 目标

把节点恢复、恢复预览、成本统计、质量统计这些新语义纳入自动化回归。

### 重点内容

- recovery service / router 契约测试
- recover -> run -> diagnostics 集成测试
- 成本统计口径测试
- 质量统计口径测试
- 最小 smoke / E2E 恢复链路回归

### 交付物

- 一组覆盖恢复主链路的自动化测试
- 一组覆盖观测统计口径的自动化测试
- 对应的验收清单

### 优先级

`P0`

### 预计收益

- 防止恢复逻辑在后续迭代中静默退化
- 防止统计口径漂移导致图表和摘要失真
- 为第三期趋势与复盘功能降低风险

---

## R6. 统计与聚合层适度整理

### 目标

在不做重迁移的前提下，适度整理与恢复、成本、质量相关的聚合层，避免后续趋势分析阶段再次大面积返工。

### 重点内容

- 收敛 run 级聚合逻辑位置
- 避免同一统计口径在多个页面重复计算
- 为 workflow metrics / run metrics 预留统一入口
- 对长期保留字段和临时兼容字段做整理

### 交付物

- 轻量聚合层整理
- 统计口径说明
- 必要的低风险重构测试

### 优先级

`P2`

### 预计收益

- 后续趋势分析更容易落地
- 页面间统计口径更一致
- 控制服务层复杂度增长

---

## 六、本期推荐开发顺序

建议按以下顺序推进：

1. `R1` 节点级恢复能力
2. `R2` 恢复预览与风险提示
3. `R5` 恢复与观测测试补强（恢复部分应伴随 `R1/R2` 同步推进）
4. `R3` 成本观测能力
5. `R4` 质量观测能力
6. `R6` 统计与聚合层适度整理

关键路径如下：

```text
R1 -> R2 -> R5
  \-> R3
  \-> R4
R6 可伴随进行，但不应阻塞主链路
```

---

## 七、本期价值

如果这一期按上述方向完成，OpenAgents 会从“主链路已经稳定的控制台”进一步进入“具备恢复与观测能力的工作台”阶段。

这期最大的价值不在于多几个页面，而在于三点：

1. 失败后恢复成本明显下降
2. 成本和质量开始具备统一观测口径
3. 第三期的趋势分析、版本差异和复盘增强有了可靠基础
